## 2. 背景与目标

### 2.1 问题陈述

- 冷启动与持续运营需要 **可传播、可验证** 的营销与社区活动，若全部堆进 **购物车与 SAP 合约路径**，迭代成本高、用户易混淆「活动资产」与「主代币」。
- Bags 提供 **标准化的 Solana 侧 launch / Intent / API**，适合作为 **增长与创作者向实验** 的执行面。

### 2.2 产品目标

1. **职权分离**：活动与营销 **不改动** 主支付合约设计原则；用户心智上 **结账 ≠ 参加活动**。
2. **快速上线**：运营可在约束模板内配置活动（链接、说明、时间窗），尽快触达 **拉新与转化**。
3. **可演进**：先 **跳转 + 凭证叙事**，再迭代 **后端记账、快照权益、与 SAP 激励的合规衔接**（若有）。

4. **入口统一在 Rewards**：**所有** 活动、奖励、Bags 相关说明与跳转 **仅** 在 DApp 的 **`/rewards`（`rewards` 模块，产品侧常称「生态活动」）** 内聚合；**`marketplace`（商城）** 专注 **选品与下单**，**不作为** 活动/奖励主入口（不在商场首页堆 Banner/活动位干扰购物动线）。

### 2.3 非目标（本期明确不做或单独立项）

- 用 Bags 代币 **直接抵扣** 商城购物车标价（跨链计价与退款复杂度极高）。
- 将 **SAP 迁移** 至 Solana 或由 Bags **替代 SAP 治理**。
- 在无明确定价与合规路径前，承诺 **BSAP ⇄ SAP** 自动兑换。

## 3. 集成逻辑（架构）

### 3.1 分层模型

```text
┌─────────────────────────────────────────────────────────┐
│                    Sapphire Mall DApp                     │
│  ┌──────────────────┐ ┌──────────────────┐ ┌──────────────────┐ │
│  │ Marketplace      │ │ Exchange         │ │ DAO / Help       │ │
│  │ `/marketplace`   │ │ `/exchange`      │ │ `/dao` · `/help` │ │
│  └──────────────────┘ └──────────────────┘ └──────────────────┘ │
│  ┌─────────────────────────────────────────┐            │
│  │ Rewards（/rewards）— 活动·奖励·Bags 唯一聚合入口      │  ← 编排层 │
│  └─────────────────────────────────────────┘            │
└───────────────────────────┬─────────────────────────────┘
                            │ HTTPS / 深链
                            ▼
┌─────────────────────────────────────────────────────────┐
│ Bags（Solana）— launch / Intent / Trade / API           │
│ 活动代币、持有者快照、创作者募资叙事等                      │
└─────────────────────────────────────────────────────────┘
```

- **编排层**：**仅** 在 **`/rewards` 模块** 内完成活动/奖励的 **列表、详情、规则、外链至 Bags**；全局顶栏 **「生态活动」** 导航进入该模块。`marketplace` **不承载** 活动主入口。可选调用 **后端** 记录「用户 ID + 钱包绑定 + 参与动作」（防刷与客服）。
- **执行层**：链上发行、交易、Intent **由 Bags 与用户 Solana 钱包** 完成；**API Key 仅存服务端**，严禁前端暴露。

### 3.2 身份与钱包

| 场景 | 钱包 | 说明 |
|------|------|------|
| 商城下单 / SAP（EVM） | EVM 钱包（现有 Wagmi 链路） | 不变 |
| Bags 活动 | Solana 钱包 | 链上动作在 **Bags 站点**完成时，由 Bags 引导连接/授权；可选「绑定 EVM 地址」用于 **同一用户画像**（仅存后端映射表，链上不强制） |

#### 3.2.1 用户分层与使用成本（产品口径）

- **仅购物 / 主链路用户**：只需 **EVM 钱包**；**不参与** Bags 活动则 **无需**安装或使用 Solana 钱包。
- **仅参与生态活动用户**：可从 **`/rewards`** 阅读规则并 **跳转 Bags**；链上执行在 Bags 侧完成，**只需满足当期规则所需的 Solana 环境**，**不强制**先在主站连接 EVM。
- **购物且需领取「与活动关联的站内权益」用户（二期 F6）**：在 **自愿**前提下完成 **EVM ↔ Solana 地址绑定**；绑定用于 **站内记账与权益**，**不能替代**用户在 Bags / Solana 上对具体交易的签名授权。

**底线表述**：不把「EVM + Solana 双钱包」写成 **进入 DApp 或购物**的前提；商城与全局主叙事 **默认仅强调 EVM**，Solana 与 Bags **仅在生态活动语境**（`/rewards`、帮助 FAQ）中展开说明。

#### 3.2.2 Bags 外链后的钱包授权

用户从 DApp **跳转至 Bags** 后，凡涉及 **Solana 链上交互**（如连接钱包、参与 launch、交换等），**通常仍需**在 Bags 站内 **连接 Solana 钱包并按步骤确认**；该步骤 **无法**由主站已连接的 **EVM 钱包**在密码学上代替。同一浏览器若曾授权过 Bags，**可能缩短重复「连接钱包」的路径**，但关键链上操作仍可能出现 **再次签名**。

**对用户说明的要点**：外链的目的之一是 **把 Solana 相关交互集中在 Bags 站点**，避免「在同一页面误以为连过 EVM 即已完成活动侧授权」；**并非**省略链上必要确认。

#### 3.2.3 绑定与顶栏引导（实施口径）

- **顶栏**：维持 **单一主连接（EVM）**；避免在用户 **刚完成 EVM 登录**时全局弹窗强制「再绑 Solana」。
- **`/rewards`**：通过 **页面说明与可选轻提示**传达「活动执行面在 Solana / Bags」；**自愿绑定**（F6）放在 **领取关联权益**、**账户或设置**等场景触发。
- **降低摩擦**：帮助文档可提示用户选用 **同时支持 EVM 与 Solana 的钱包客户端**（仍为多条链地址，但可减少多个浏览器插件的安装成本）。

#### 3.2.4 用户可见文案（与前端对齐）

DApp 中 **`rewards` 模块**的用户文案以仓库 **`web_client/sapmall-dapp/src/i18n/locales/*/translation.json`** 的 **`rewards`** 命名空间为准；变更口径时 **同步更新 PRD 本节索引与 JSON**，避免设计与实现分叉。

### 3.3 数据流（概念）

1. 用户在 **`/rewards` 活动/奖励** 页点击 **「前往 Bags / 参与活动」** → 打开 **Launch Intent URL** 或 Bags 项目页（URL 由运营配置 + 服务端校验域名白名单）。
2. 用户在 Solana 侧完成操作 → 链上状态由 Bags / 浏览器可查。
3. （可选）后端 **Webhook / 定时任务** 拉取公开数据或接收运营导入快照 → **标记用户权益**（如白名单、站内徽章），**不与订单表混写**。

## 4. 功能范围

### 4.1 MVP（建议第一期）

| ID | 功能 | 说明 | 与现有模块结合 |
|----|------|------|----------------|
| F1 | **活动列表与详情** | 标题、时间、规则摘要、风控提示、官方链接 | **`sapmall-dapp` `src/pages/rewards/`（路由 `/rewards`）** — 所有活动与奖励 UI **仅此模块** 承载，扩展为列表 + 详情 |
| F2 | **Launch Intent 外链** | 一键跳转预填充 Intent | 运营在后台或配置表维护 URL；前端只做展示与跳转 |
| F3 | **Bags 项目 / Token 展示卡** | 展示 mint、名称、链接（只读） | 卡片组件读取 **服务端下发 JSON**（避免前端写死） |
| F4 | **规则与边界文案** | SAP ≠ 活动代币、非投资建议 | 与 README / `#governance` 口径一致 |
| F5 | **服务端 API Key 调用（可选）** | 拉取公开元数据（不含密钥下发前端） | `backend_service` 新增最小接口（若本期只做静态配置则可延后） |

### 4.2 第二期（增强）

| ID | 功能 | 说明 |
|----|------|------|
| F6 | **EVM ↔ Solana 地址绑定** | 用户自愿绑定，用于后续权益发放 |
| F7 | **快照导入与权益兑换** | 运营上传快照 CSV → 系统发兑换码或站内权益 |
| F8 | **Admin 活动配置台** | CRUD 活动、上下线、关联 Bags 链接 |
| F9 | **埋点与漏斗** | **`/rewards` 模块** PV → 外链点击 → （若有）注册/首单 |

### 4.3 明确依赖 Bags 平台的能力映射

| Bags 能力（概念） | 在本项目中的用途 |
|-------------------|------------------|
| Launch | 活动代币 / 项目冷启动 |
| Launch Intent | 拉新转化短链路 |
| API / SDK | 服务端安全调用、未来自动化生成 Intent |
| Trade / Fees（若采用） | 交易周、分润类实验（单独合规评审） |

## 5. 与现有功能的结合方式

| 现有模块 | 结合点 | 原则 |
|----------|--------|------|
| **商城 Marketplace**（`marketplace`） | **不承担** 活动/奖励主入口；购物路径 **不包含** 商场首页活动 Banner | 用户参加活动统一走顶栏 **生态活动 → `/rewards`**；商城 **仅** 货架与下单，**不改变商品价格逻辑** |
| **兑换 Exchange** | 保持 SAP/稳定币叙事；不与 BSAP 混排为「同一种余额」 | 若需提示「参与活动」，在 **`/rewards`** 说明或通过顶栏进入 rewards，**不在** 兑换页替代 rewards |
| **DAO / 社区参与** | 治理讨论可 **引用** 活动规则；**子预算投票**若存在须单独 PRD | 核心治理仍以 SAP / EVM 文档为准 |
| **帮助 Help** | FAQ：**活动与主支付区别**、**谁需要哪种钱包**、**跳转 Bags 后为何仍需连接 Solana**、防骗与官方链接 | 降低客服压力与「双钱包」误解 |
| **管理后台 Admin** | 活动配置、链接审核、快照导入（二期） | 权限角色：运营 / 管理员 |
| **SAP / Tokenomics** | 文档交叉引用，明确 **活动代币定位** | 避免二级市场误读 |

## 6. 场景案例（端到端）：「创作者季 × BSAP」

> 虚构但可落地；实施时替换为真实链接、ticker 与时间。

### 6.1 业务故事

平台发起 **「蓝宝石创作者季」**：鼓励创作者与早期用户参与 **Bags 侧 launch**，形成 **链上参与凭证（BSAP）**，用于后续 **站内权益**（如优先审核通道、活动纪念品），**不参与购物车折扣**。

### 6.2 角色

- **Alice**：新用户，仅有 Solana 钱包想参与活动。
- **Bob**：老用户，有 EVM 钱包购物，希望绑定身份领权益。
- **运营**：配置活动与 Intent 链接。

### 6.3 用户旅程

1. **触达**：Bob 打开 DApp → 顶栏点击 **生态活动**（进入 **`/rewards`**）→ 在活动/奖励列表中选择「创作者季」→ 打开详情页。（**不在** `marketplace` 商场首页放置活动 Banner 作为主路径。）
2. **知情**：详情页展示 **规则摘要、风险提示、SAP 与 BSAP 边界** → 勾选「已阅读」→ 点击 **「前往 Bags 参与」**。
3. **链上执行**：跳转 **Launch Intent URL** → 连接 Phantom → 完成 Bags 侧规定动作（launch 参与/领取等，以当期规则为准）。
4. **回流（可选 MVP 之后）**：Bob 在 **账户设置** 绑定 Solana 地址与 EVM 地址 → 后端标记 **「创作者季参与者」** → DAO 页或帮助页展示 **权益说明**。
5. **购物路径**：Bob 照常 **EVM 下单**，订单金额 **不使用 BSAP 抵扣**；客服不支持「用 BSAP 退款购物车」类诉求。

### 6.4 系统交互（简序）

```text
DApp /rewards 活动详情 ──HTTPS──> Bags 页面 / Intent
       │
       └──（可选）Backend 记录 click_id + user_id + timestamp

Solana 链上状态 ◄──用户钱包──► Bags 程序/API

Backend（可选）◄──快照/导入── 运营 → 权益表（与 orders 表隔离）
```

### 6.5 成功指标（示例）

- **`/rewards` 模块** UV、外链 CTR、完成 Bags 侧参与人数（若能统计）、后续 **首访商城注册率**（若有归因）。
- **负面指标**：混淆 SAP/BSAP 的工单量 → 应用 FAQ 优化。

## 7. 安全与合规要点

- **链接白名单**：仅允许 `bags.fm`、`dev.bags.fm`、`docs.bags.fm` 等官方域名（运营新增须审批）。
- **密钥**：`x-api-key` 仅服务端；日志脱敏。
- **文案**：活动页必备 **非投资建议、实验性质、地区限制占位**（法务修订）。
- **防钓鱼**：与 Discord `#start-here` 口径一致，**不私信索要助记词**。

---

## 8. 验收标准（MVP）

1. 用户可在 **`/rewards`（rewards 模块）** 查看至少 **1 个** 上线活动及详情；**商场首页（marketplace）无活动主入口依赖**。
2. 详情页可 **一键跳转** 至有效 Intent / Bags 页面（测试环境可接 Bags dev）。
3. 全站关键路径可见 **「活动 ≠ 结账」** 说明。
4. 无前端暴露 Bags API Key；无购物车引入 BSAP 计价逻辑。
